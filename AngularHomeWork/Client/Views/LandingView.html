Features

Schools

  • Each school will host the app and database from their own servers.
  • Allowing the Schools to have more oversite over their own data.
  • As the program is read heavy, having each school's data on it's own database results in scalability being far less of an issue.

Teachers

  • Create and delete classrooms.
  • Create, edit, and delete assignments, inside each classroom.
  • Can view which students have marked an assignment as done.

Students

  • Subscribe to classrooms.
  • See the assignments that they have been set for each classroom.
  • Mark assignmemts as done.

About the App

  • My Objective

    The choice of technologies was driven by the following design objectives.

    1. Responsiveness - AJAX rather than ASP.net forms.
    2. Security - The built in security features of both AngularJs (Html encoding) and Asp.net MVC (Web Forms, Role Authorisation).
    3. Clean consistent look - Material Design.
    4. Separation of code and graphics in the front end - AngularJs.
    5. Maintainability - use of MVC pattern on server and client.
    6. Multiple client platforms - RESTful interface.

  • Problems encountered

    The biggest problem that I encountered was in deciding which architecture to use. Initially I tried to build the whole thing using only MVC controllers and the standard Asp.Net MVC pattern. However I found that this meant that each button click would mean loading up a whole new angularJs app, not only was this slow and clunky, it also seemed like a wast of AngularJs. So instead I opted to use the Asp.net MVC controllers to navigate to 3 single page AngularJs apps (Login, Teacher, and Student), loaded with some initial data through Razor. Then all subsequent server calls would be done with Ajax routed through Asp.Net Web Api controllers.

  • Security

    Hashing and salting user passwords, encrypted with SHA256

    Web Forms Authentification via login details held on data base.

    User Roles Authorisation

    Html encoding built into AngularJs to avoid cross-site-scripting attacks.

    Fully parameterised SQL to avoid SQL injection.

    Stopping cookie theft, via script attack, by adding an 'HttpOnly' flag to my web.config file.
